# DB Entity Brainstorm

엔티티 설계를 함께 검토하고 방향을 잡아주는 브레인스토밍 스킬이다. **코드를 수정하지 않는다 — 의견만 제공한다.**

## Step 0: 입력 확인

사용자가 다음 두 가지를 제공해야 한다:

1. **엔티티 구조** — 파일 경로, 코드 블록, 또는 컬럼 나열
2. **의미/도메인 규칙** — "user가 SNS를 1개만 가진다", "주문은 여러 상품을 가진다", "가격 이력을 보존해야 한다" 등

둘 중 하나라도 빠지면 먼저 물어본다. 의미 없이 구조만 보면 정답이 없다.

## Step 1: 컨텍스트 수집

- 입력으로 받은 엔티티 파일을 Read로 읽는다
- 관련된 다른 엔티티(FK 대상, OneToOne/ManyToMany 상대편)가 있으면 함께 읽는다
- 같은 도메인의 enum/DTO가 있으면 훑는다
- 주변 마이그레이션이나 동일 패턴의 다른 엔티티(이 프로젝트의 컨벤션 파악용)도 짧게 본다

이 단계에서 절대 결론을 내지 않는다. 사실 수집만 한다.

## Step 2: 세 가지 관점으로 정리

다음 형식으로 보고한다. 각 항목은 **불릿 + 근거** 한 쌍으로 짧게 쓴다. 길게 쓰지 않는다.

```
## 1. 괜찮은 점
- 무엇이 잘 되어 있는지 (현재 설계의 장점)
- 왜 그것이 장점인지 (도메인 규칙·성능·일관성 측면 근거)

## 2. 대중이 많이 선택한 방식
- 같은 요구사항에서 업계가 일반적으로 채택하는 패턴
- 그 패턴이 보편적인 이유 (확장성, 표준 ORM 패턴, 마이그레이션 안전성 등)
- 현재 설계와의 차이점

## 3. 내 추천 방향
- 이 프로젝트 컨텍스트에서 권하는 방향 (대중 패턴 그대로일 수도, 다를 수도 있음)
- 그렇게 추천하는 이유 (요구사항 + 프로젝트 코드 컨벤션 + YAGNI 균형)
- 지금은 안 해도 되지만 "이런 신호가 보이면 그때 바꿔라"는 트리거 조건
```

## 작성 원칙

- **YAGNI 우선**: 지금 요구사항에 안 쓰이는 확장성은 권장하지 않는다. "혹시 모르니" 분리는 오버엔지니어링.
- **트레이드오프를 명시**: "X가 좋다"가 아니라 "X는 ~를 얻는 대신 ~를 잃는다"로 쓴다.
- **대중 패턴이 항상 정답은 아님**: 2번과 3번이 갈려도 괜찮다. 갈리면 왜 갈리는지 설명한다.
- **카디널리티/NULL/Unique를 항상 점검**: 1:1, 1:N, N:M 중 어느 것인지, NULL 허용 여부, 복합 unique가 필요한지.
- **이 프로젝트의 컨벤션을 따른다**: TypeORM + SnakeNamingStrategy + `CommonEntity` 상속 패턴 등 기존 코드 스타일을 존중.
- **확신이 없으면 솔직하게 말한다**: "도메인 규칙을 더 알아야 판단 가능"이라고 명시하고 후속 질문을 한다.

## 자주 다루는 판단 포인트

- **인라인 컬럼 vs 별도 테이블**: 1:1이면 인라인이 단순. 1:N이거나 메타데이터/이력이 필요하면 분리.
- **Enum vs 참조 테이블**: 값이 안 변하고 적으면 enum. 운영자가 추가/수정해야 하면 테이블.
- **Soft delete vs Hard delete**: 감사·복구 요구가 있으면 soft, 없으면 hard.
- **Unique 제약**: 비즈니스 규칙상 중복 불가인데 DB에 막혀있지 않은 경우를 항상 본다 (partial unique index 포함).
- **NULL 일관성**: "셋 다 NULL이거나 셋 다 NOT NULL" 같은 동시 제약은 CHECK constraint로 보강 가능.
- **인덱스**: 자주 조회되는 컬럼/조합에 인덱스가 있는지. FK는 자동이지만 복합 조회는 별도.
- **JSON 컬럼**: 스키마가 자주 바뀌거나 sparse하면 JSON, 안정적이면 정규화.

## Step 3: 마무리

세 관점을 다 제시한 뒤, 사용자가 어느 방향으로 갈지 묻는다. **사용자 선택 전에 코드를 수정하지 않는다.** 사용자가 방향을 정하면 그때 별도 작업으로 진행하거나 다른 스킬(예: 코드 작성)로 넘어간다.
